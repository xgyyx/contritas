# Contritas — Deep Research Agent 需求文档

> 通过反证逼近真相的通用深度研究 Agent。

---

## 一、产品定位

Contritas 是一个**结构化尽职调查 Agent**。它接收用户提出的决策命题（商业方案、技术决策、政策判断、投资论点等），将其拆解为可验证的核心假设，通过多源检索和交叉验证，输出带置信度标注的专业尽调报告。

**一句话定义**：不证明你是对的，证明你经得起尽调。

### 目标用户

独立创业者、小团队负责人、早期投资人——他们面临真实的决策风险，但请不起专业尽调团队。Contritas 让个人也能获得结构化、有方法论保障的尽职调查。

### 与现有方案的差异

| 方案                     | 定位                             | Contritas 的差异                                                    |
| ------------------------ | -------------------------------- | ------------------------------------------------------------------- |
| Perplexity Deep Research | 搜索增强问答——用户问什么答什么   | 内建反证流程：强制从反面质疑每个假设，不依赖用户 prompt 技巧        |
| ChatGPT Deep Research    | 综合研究助手——倾向正面论证       | 证据分级 + 交叉验证作为硬约束，不是可选步骤                         |
| 手动 prompt "请反驳我"   | 依赖用户提示工程能力，质量不稳定 | 方法论内建：假设拆解→反向质疑→多源验证→置信度评分，流程固定且可审计 |

**核心差异总结**：Perplexity/ChatGPT 是"搜索增强问答"，Contritas 是"结构化尽调流程"。差异不在能力，在于**承诺**——保证不放过你的盲点。

## 二、核心原则

1. **审计口径，非鼓励口径** — 不迎合用户预期，不只找支持性证据
2. **反证优先** — 每个假设先问"如果它不成立，会因为什么"
3. **证据分级** — 区分高可信来源与弱证据，标注置信度
4. **方法论通用，证据来源领域化** — 尽调方法论不限行业，但证据来源优先选取目标领域的权威渠道（如商业尽调优先工商/SEC/天眼查，技术尽调优先 benchmark/RFC/官方文档）
5. **结论先行** — 报告开头给判断，读者无需通读全文即可获得核心结论
6. **深度优先于广度** — 当资源受限时，优先保证高权重维度的证据深度（更多来源、更充分验证），允许低权重维度降至最低门槛（3 个来源）

## 三、用户故事

### 主场景

> 作为一个决策者，我有一个判断/方案/想法，我想知道它是否真的成立、哪些前提最脆弱、主要风险是什么，以便我做出更好的决策。

### 典型输入示例

- "我想做一个面向中国开发者的出海模板+社群产品，月入 5 万可行吗？"
- "我们团队打算用 Rust 重写核心服务，值得吗？"
- "2026 年下半年美联储大概率降息，这个判断靠谱吗？"
- "我打算 all-in AI agent 赛道创业，风险有多大？"

### 典型输出

一份结构化深度研究报告（详见第六节）。

## 四、功能需求

### 4.1 假设拆解

- 从用户输入中提取 3-8 个核心假设
- 按"如果此假设不成立，对结论影响多大"排序
- 识别哪些是事实性假设（可查证）、哪些是判断性假设（需推理）

### 4.2 动态维度生成

- 根据主题类型自动选择研究维度
- 不使用固定模板；维度必须从输入推导
- 每个维度包含：名称、核心问题、反向质疑

### 4.3 多源证据检索

- 为每个维度执行多轮检索
- 支持中英文双语搜索
- 优先使用高可信来源：官方文档、法规、权威统计、一手数据、学术报告
- 必须读取页面内容提取关键数据，不能只依赖搜索摘要
- 每个维度目标 5 个独立来源（最低 3 个），其中至少 2 个为高可信来源。低于 3 个时必须标注并说明缺失原因

### 4.4 证据质量评估

- 为每条证据标注可信等级（高 / 中 / 弱）
- 标注证据的时效性（发布日期）
- 标注证据与假设的关系（支持 / 削弱 / 限定）
- 识别仍缺失的关键证据

### 4.5 交叉验证

- 同一假设的多条证据之间是否一致
- 不同来源是否存在矛盾
- 如有矛盾，分析可能原因并标注

### 4.6 定量测算（如适用）

- 当用户输入包含数字型假设（收入目标、市场规模、成本预期等）时
- 自动进行 sanity check：反推所需条件是否合理
- 输出测算表格

### 4.7 综合判断与评分

- 每个维度给出结论标签：✅支持 / ⚠️存疑 / ❌不支持
- 给出总体置信度或评分
- 解释"为什么不是更高"和"为什么不是更低"

### 4.8 行动建议

- 如果继续推进：应该怎么做、先验证什么
- 如果暂缓：需要补充什么证据
- 如果否定：核心原因是什么
- 如果需要修正：给出替代方向

### 4.9 引用管理

- 每个来源自动编号
- 行文中插入引用标记
- 报告末尾生成参考来源列表（含 URL 和摘要）

### 4.10 输入预处理与边界场景

**场景 A：输入过于模糊**

当用户输入不包含可拆解的具体命题时（如"帮我研究一下 AI"），Agent 必须：

1. 向用户追问：你具体想验证什么判断？想做什么决策？
2. 提供 2-3 个可能的研究方向供用户选择
3. 不得在无明确命题的情况下启动研究流程

**场景 B：输入是纯观点而非可验证命题**

如"我觉得 React 比 Vue 好"，Agent 应：

1. 将其转化为可验证形式：在什么场景下、按什么标准、React 是否优于 Vue？
2. 向用户确认转化后的命题是否符合其意图

**场景 C：输入包含多个独立主题**

Agent 应：

1. 识别并列出各独立主题
2. 建议用户选择最优先的 1-2 个进行深入研究
3. 如用户坚持全做，按优先级顺序依次完成

**场景 D：输入包含明显错误前提**

如"既然中国人口已经 20 亿了..."，Agent 应：

1. 在假设拆解阶段直接标注该前提与事实不符
2. 提供正确数据
3. 询问用户是否在修正前提后继续研究

### 4.11 多轮交互规范

Agent 在研究过程中**可以且应该**向用户追问的场景：

- 输入模糊，无法提取可验证假设时（必须追问）
- 发现用户隐含的关键假设需要确认时（建议追问）
- 研究中出现两条截然不同的路径，需要用户选择优先级时（建议追问）

Agent **不应该**追问的场景：

- 已有足够信息启动研究时
- 追问的内容本身可以通过检索获得时
- 用户已明确表示"你自己判断"时

### 4.12 检索失败与降级策略

当某个维度无法获得足够高可信来源时：

| 情况                               | 处理方式                                                                     |
| ---------------------------------- | ---------------------------------------------------------------------------- |
| 搜索返回 0 结果                    | 换关键词、换语言、换搜索引擎重试，最多 3 轮                                  |
| 只能找到弱证据（自媒体、单一来源） | 在报告中明确标注"证据强度：弱"，并说明缺失原因                               |
| 页面无法访问                       | 尝试缓存/存档版本（web archive），失败则跳过并记录                           |
| 某维度完全无法获得有效证据         | 在报告中保留该维度，结论标注为"⚠️无法验证——缺乏可获取的公开证据"，不删除维度 |

降级底线：任何维度不得因为"找不到证据"而被静默删除。缺失本身就是重要信息。

### 4.13 评分机制

**评分区间定义：**

| 分数区间   | 含义                                                         |
| ---------- | ------------------------------------------------------------ |
| 8.0 - 10.0 | 核心假设大部分被高可信证据支持，风险可控，建议推进           |
| 6.0 - 7.9  | 部分假设成立，但存在明确风险或不确定性，需要补充验证后再决策 |
| 4.0 - 5.9  | 多个关键假设存疑或不成立，方案需要重大修正才有可行性         |
| 2.0 - 3.9  | 核心假设大部分不成立或证据严重不足，不建议按当前方案推进     |
| 0.0 - 1.9  | 方案与现实严重脱节，建议放弃或完全重构方向                   |

**评分规则：**

1. 总分不是各维度简单平均。Agent 应根据维度对最终结论的影响力进行加权
2. 一票否决规则：仅当以下情况成立时，即使其他维度全部支持，总分也不应超过 4.0：
   - 方案本身违反现行法律法规（已确认，非存疑）
   - 核心技术依赖已被证明物理上/原理上不可行
   - 关键前提与已确认事实直接矛盾（如市场已不存在）
   - 边界情况（如"在当前团队条件下不可行"）应标注为 ⚠️ 高风险，但不触发一票否决
3. 评分必须附带一段解释：为什么是这个分数而非更高或更低
4. Agent 不应追求精确到小数点后一位的伪精度。如果无法区分 5.2 和 5.5，应给出区间（如 5.0-5.5）

**维度权重确定方式：**

- 不预设固定权重
- Agent 在报告"综合评估"部分说明各维度对本次研究的相对重要性
- 重要性判断标准：如果该维度结论翻转，是否会改变总体建议？如果会，则为高权重维度

### 4.14 输出长度自适应规则

| 主题复杂度 | 判断标准                                                   | 核心报告长度  | 含证据附录    |
| ---------- | ---------------------------------------------------------- | ------------- | ------------- |
| 低         | 假设 ≤ 3 个，维度 ≤ 3 个，无定量测算需求                   | 3000-4000 字  | ≤ 6000 字     |
| 中         | 假设 4-6 个，维度 4-5 个，可能有定量测算                   | 5000-7000 字  | ≤ 10000 字    |
| 高         | 假设 ≥ 6 个，维度 ≥ 6 个，有多项定量测算，涉及多个交叉领域 | 7000-10000 字 | ≤ 15000 字    |

判断复杂度的时机：Phase 2（规划阶段）完成后，Agent 确定维度数量时同步确定目标长度。

### 4.15 报告迭代与深挖

用户可能对首版报告提出以下后续请求：

| 请求类型                           | Agent 响应方式                                                                           |
| ---------------------------------- | ---------------------------------------------------------------------------------------- |
| "维度 X 再深入一些"                | 对该维度追加 2-3 轮检索，扩展证据，更新该维度结论，不重写其他维度                        |
| "我不同意维度 Y 的结论，理由是..." | 将用户理由纳入考量，重新检索验证，如证据支持用户观点则修正结论，否则保持原判断并解释原因 |
| "漏了一个重要角度 Z"               | 新增维度 Z，执行完整的检索→分析→结论流程，追加到报告中，并更新综合评分                   |
| "给我一个精简版 / 摘要版"          | 输出仅包含：结论先行 + 假设拆解表 + 综合评估表 + 建议，省略详细论证                      |
| "基于这份报告，帮我写行动计划"     | 基于报告建议部分，输出具体的时间线、检查点和下一步行动清单                               |

## 五、非功能需求

| 项目             | 要求                                                                                                                                  |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 单次报告完成时间 | 按复杂度分层（见下表），不设硬性上限                                                                                                  |
| 证据来源数量     | 低复杂度 ≥ 15 个、中复杂度 ≥ 25 个、高复杂度 ≥ 40 个独立来源（整份报告合计）                                                          |
| 输出长度         | 3000-15000 字（根据 4.14 节复杂度规则自适应；核心报告 ≤ 10000 字，证据附录不限）                                                      |
| 语言             | 输出语言跟随用户输入语言；如用户用中文提问，报告用中文输出，但证据摘录保留原文语言                                                    |
| 结论一致性       | 相同输入在 7 天内多次执行，总体判断方向（推进/暂缓/否定）应一致，评分浮动不超过 ±1.5 分。允许因新证据出现而改变结论，但需标注变化原因 |
| 可解释性         | 每个结论必须可追溯到具体证据编号；报告中不得出现无来源支撑的断言                                                                      |
| 可审计性         | 参考来源表必须包含可访问的 URL，读者可自行验证                                                                                        |

**报告完成时间分层：**

| 复杂度 | 预计时长   | 搜索调用次数 | 独立来源数 |
| ------ | ---------- | ------------ | ---------- |
| 低     | 10-15 分钟 | 30-50 次     | 15-20      |
| 中     | 20-30 分钟 | 50-80 次     | 25-40      |
| 高     | 30-60 分钟 | 80-150 次    | 40-60+     |

### 5.1 Agent 所需工具能力清单

Contritas 的运行依赖以下工具能力。具体实现可以是 API、SDK 或平台内置功能，但必须覆盖以下类别：

| 能力类别             | 用途                       | 最低要求                                                        |
| -------------------- | -------------------------- | --------------------------------------------------------------- |
| **Web 搜索**         | 为每个维度检索公开信息     | 支持自然语言查询；支持中英文；单次研究最高可执行 150 次搜索调用 |
| **网页内容提取**     | 读取搜索结果页面的完整内容 | 能获取页面正文文本；能处理常见反爬（JS 渲染页面可降级跳过）     |
| **算术与表格生成**   | 定量测算、sanity check     | 四则运算；百分比计算；生成 Markdown 表格                        |
| **结构化输出**       | 按模板格式化最终报告       | 支持 Markdown 输出；支持表格、列表、多级标题                    |
| **状态管理（可选）** | 断点续做、多轮迭代         | 能记住前一次报告内容，在用户要求深挖时增量更新而非重做          |
| **多语言处理**       | 检索和输出均需支持中英文   | 能用中文关键词搜索中文来源；能用英文关键词搜索英文来源          |

**不需要的能力：**

- 不需要代码执行环境（测算仅涉及基础算术）
- 不需要数据库
- 不需要图片/视频处理
- 不需要实时数据流接入

### 6.2 输出示例片段

以下为"结论先行"部分的一个填充示例，展示预期的具体程度：

```markdown
## 一、结论先行

- **总体判断**：该技术方案可行但收益被高估，建议缩小范围后再推进
- **综合评分**：5.5-6.0 / 10
- **关键发现**：
  1. 核心性能瓶颈确实存在，但根因不是语言选择而是架构设计
  2. Rust 重写的工程成本被低估约 3 倍（团队无 Rust 经验）
  3. 渐进式重写关键热路径可获得 80% 收益，仅需 20% 投入
- **最大风险**：团队 Rust 学习曲线导致项目延期超过 6 个月
- **核心建议**：先用 2 周做单模块 PoC 验证性能假设，再决定是否全面重写
```

## 七、Agent 工作流

### Phase 0：输入验证

| 项目         | 说明                                         |
| ------------ | -------------------------------------------- |
| **输入**     | 用户原始文本                                 |
| **输出**     | 通过验证的研究命题，或向用户发出的追问       |
| **判断条件** | 输入是否包含至少一个可拆解的具体命题？       |
| **通过**     | 进入 Phase 1                                 |
| **不通过**   | 按 4.10 节边界场景处理，向用户追问后等待回复 |

### Phase 1：理解与拆解

| 项目         | 说明                                                          |
| ------------ | ------------------------------------------------------------- |
| **输入**     | 验证通过的用户命题                                            |
| **输出**     | 假设列表（3-8 个），每个标注类型（事实性/判断性）和重要性排序 |
| **完成标准** | 至少提取出 3 个假设，且最重要的假设被标注为"高"               |
| **异常处理** | 如果只能提取 1-2 个假设，说明主题聚焦度高，按低复杂度处理     |

### Phase 2：规划

| 项目         | 说明                                                                                                                          |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| **输入**     | 假设列表                                                                                                                      |
| **输出**     | 研究计划，包含：维度列表（每个含名称+核心问题+反向质疑）、每个维度的检索关键词（中英文各 2-3 组）、目标来源类型、预估输出长度 |
| **完成标准** | 维度数量 = 3-8 个；每个维度至少有 2 组检索关键词                                                                              |
| **异常处理** | 如果某假设无法对应任何明确维度，将其归入"综合判断"而非强行创建维度                                                            |

### Phase 3：检索与分析

| 项目                   | 说明                                                                                                                                                                                         |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **输入**               | 研究计划中的维度列表和检索关键词                                                                                                                                                             |
| **输出**               | 每个维度的证据集合（每条含：来源、URL、可信度、日期、关键摘录、与假设关系）                                                                                                                  |
| **执行方式**           | 各维度逻辑独立，实现时可并行亦可串行，不影响结果正确性                                                                                                                                       |
| **每个维度的检索流程** | 第 1 轮：用预设关键词搜索 → 提取页面内容 → 标注证据质量。第 2 轮：基于第 1 轮结果调整关键词（如发现新术语、新角度）→ 补充搜索。第 3-5 轮（按需）：仅当前序轮次后仍未满足来源数量要求时触发    |
| **追加检索触发条件**   | ① 当前维度证据不足 3 条独立来源或不足 2 条高可信来源（目标 5 条）；② 发现与预期相反的关键信息需要交叉验证；③ 出现预期外的新维度（此时新增维度并补充检索）                                   |
| **终止条件**           | 每个维度最多 5 轮检索。5 轮后仍不足则标注"证据不足"进入下一阶段                                                                                                                              |
| **异常处理**           | 搜索 API 超时：重试 1 次后跳过该查询，记录失败。页面无法访问：尝试 web archive，失败则跳过并记录                                                                                             |

### Phase 4：验证与测算

| 项目                 | 说明                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------ |
| **输入**             | Phase 3 产出的所有维度证据集                                                               |
| **输出**             | 交叉验证结果（矛盾标注）、定量测算表格（如适用）、各维度初步结论                           |
| **交叉验证规则**     | 同一假设若有 2+ 条证据方向矛盾，必须标注矛盾并分析可能原因（来源偏见、时间差异、口径不同） |
| **定量测算触发条件** | 用户输入中包含具体数字（收入目标、成本假设、市场规模估计、时间预期等）                     |
| **完成标准**         | 每个维度都有初步结论标签（✅/⚠️/❌），所有矛盾已被标注                                     |

### Phase 5：综合与输出

| 项目         | 说明                                                                                                                                               |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **输入**     | Phase 4 的结论、证据、测算结果                                                                                                                     |
| **输出**     | 完整格式化报告（按第六节模板）                                                                                                                     |
| **评分执行** | 按 4.13 节规则计算总分，附解释                                                                                                                     |
| **质量自检** | 输出前检查：① 每个维度是否有反向质疑 ② 每个结论是否有 ≥ 3 条证据（其中至少 1 条高可信） ③ 参考来源表是否完整 ④ 总分解释是否包含"为什么不更高/更低" |
| **异常处理** | 如自检发现某维度结论缺乏证据支撑，回退到 Phase 3 对该维度补充一轮检索，最多回退 1 次                                                               |

### 工作流状态图

```
[Phase 0] ──不通过──→ 追问用户 ──收到回复──→ [Phase 0]
    │通过
    ▼
[Phase 1] ──完成──→ [Phase 2] ──完成──→ [Phase 3]
                                            │
                                    ┌───────┴───────┐
                                    │  按维度并行    │
                                    │  每维度≤3轮   │

                                                              │是

                                                              │
                                                        自检通过
                                                              │
                                                         输出报告
```

## 八、关键约束

### 8.1 必须做

- 每个维度必须包含反向质疑
- 每个结论必须有至少 3 条证据支撑（其中至少 1 条为高可信来源）
- 必须区分事实、推论和判断
- 必须标注不确定性和缺失信息
- 必须在报告中说明"为什么不是更高/更低"

### 8.2 禁止做

- 禁止只找支持性证据
- 禁止使用未注明来源的数据
- 禁止把营销材料当高可信来源
- 禁止把单个案例当作普遍性证明
- 禁止混淆不同口径的数据（如总量与子集、毛收入与净收入、一次性与经常性）
- 禁止在证据不足时给出高置信结论
- 禁止省略对用户不利的发现

### 8.3 证据使用规则

- 官方文档 > 权威统计 > 学术报告 > 行业报告 > 一手案例 > 社区讨论 > 自媒体
- 单一来源不能作为结论的唯一依据
- 超过 2 年的数据必须标注时效风险
- 有利益冲突的来源（如平台自己说自己好）必须交叉验证

## 九、成功标准

| 指标             | 目标                                                     | 衡量方式                                                                                                       |
| ---------------- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| 结论方向准确率   | 北极星指标（长期追踪）：事后验证 ≥ 70% 判断方向正确      | 用户在 30/90/180 天后回顾报告结论，标记"方向正确"或"方向错误"；统计正确率。注：V1 阶段仅收集数据，不作为硬性 KPI |
| 证据覆盖度       | 每个维度目标 ≥ 5 个独立来源（最低 3 个），其中 ≥ 2 个高可信 | 报告输出后自动统计参考来源表中每个维度对应的来源数量                                                           |
| 用户决策信心提升 | 用户阅读报告后对决策方向的确定性提升                     | 报告输出后向用户提问："读完后你对决策方向是否更清晰？"选项：更清晰 / 没变化 / 更困惑。目标：≥ 80% 选择"更清晰" |
| 假设遗漏率       | 事后发现的关键遗漏假设 ≤ 1 个                            | 用户在决策执行后回顾："报告是否遗漏了你后来发现很重要的因素？"统计遗漏数                                       |
| 建议可执行性     | 建议部分的每条行动都有明确的执行主体、时间框架和验证标准 | 逐条检查报告"建议"部分：每条是否包含"谁做"+"做什么"+"怎么判断做完了"。包含率目标 ≥ 90%                         |
| 报告完整性       | 输出报告符合第六节模板的所有必填项                       | Phase 5 自检通过率 = 100%（即每份报告都通过了质量自检）                                                        |

## 十、命名与品牌

- **名称**：Contritas
- **含义**：Contra（反向）+ Veritas（真理）= 通过反证逼近真相
- **拉丁语 contritus 另义**："碾碎" — 把假设碾碎检验
- **Slogan**：_Grind assumptions. Surface truth._
