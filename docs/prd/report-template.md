# Contritas — 报告输出模板

> 本文档定义 Contritas 深度研究报告的标准格式。Agent 生成报告时必须遵循此模板。
>
> 相关文档：[PRD](./prd.md) | [Agent 行为规范](../guides/agent-behavior.md)

---

## 一、报告模板

```markdown
# 深度研究报告：{主题}

## 一、结论先行

- **总体判断**：用一句不超过 30 字的话回答用户最初的问题
- **综合评分**：X / 10（或 X-Y / 10 区间）
- **关键发现**：
  1. {最重要的发现}
  2. {第二重要的发现}
  3. {第三重要的发现}
- **最大风险**：{如果推进，最可能导致失败的单一因素}
- **核心建议**：{一句话建议下一步行动}

## 二、研究口径

- **本报告验证什么**：{具体列出被审计的核心命题}
- **不覆盖什么**：{明确排除的范围，避免读者误以为遗漏}
- **证据截止时间**：{检索执行的日期}
- **主要证据语言**：{中文 / 英文 / 双语}

## 三、核心假设拆解

| #   | 假设内容   | 类型          | 重要性   | 判断     | 证据强度 |
| --- | ---------- | ------------- | -------- | -------- | -------- |
| 1   | {假设描述} | 事实性/判断性 | 高/中/低 | ✅/⚠️/❌ | 强/中/弱 |

## 四、分维度研究

### 维度 1：{动态名称}

#### 核心问题

{这个维度要回答的关键问题，1-2 句}

#### 反向质疑

{如果用户的假设在这个维度上不成立，最可能是因为什么？列出 2-3 个反向问题}

#### 证据与观察

- **[来源编号] {来源名称}**（可信度：高/中/弱，日期：YYYY-MM）
  - 关键数据/引述：{原文摘录或关键数据点}
  - 与假设关系：支持 / 削弱 / 限定

（每个维度目标 5 条证据，最低 3 条；其中至少 2 条来自高可信来源）

#### 分析与推论

{基于证据的逻辑推演，必须说明推理链条，标注哪些是事实、哪些是推论}

#### 阶段性结论

- **判断**：✅支持 / ⚠️存疑 / ❌不支持
- **置信度**：高/中/低
- **关键不确定性**：{本维度仍无法确认的 1-2 个要点}

（重复 3-8 个维度）

## 五、证据质量总览

### 高可信证据（官方文档、权威统计、法规）

| #   | 来源 | 用于支撑的假设/维度 |
| --- | ---- | ------------------- |

### 中等可信证据（行业报告、一手案例、社区讨论）

| #   | 来源 | 用于支撑的假设/维度 |
| --- | ---- | ------------------- |

### 仍缺失的关键信息

- {描述缺失的信息及其对结论的潜在影响}

## 六、综合评估

| 维度     | 结论     | 证据强度 | 对总体结论的权重 | 风险等级 |
| -------- | -------- | -------- | ---------------- | -------- |
| {维度名} | ✅/⚠️/❌ | 强/中/弱 | 高/中/低         | 高/中/低 |

**评分说明**：{2-3 句话解释总分为什么是这个值，为什么不是更高，为什么不是更低}

## 七、建议

### 如果推进

1. {具体行动 1}
2. {具体行动 2}
3. {优先验证什么}

### 如果暂缓

- {需要补充的证据或条件}
- {建议的等待观察点}

### 如果否定 / 重构

- {核心否定原因}
- {可能的替代方向}

## 八、参考来源

| #   | 来源名称 | URL   | 类型                    | 可信度   | 摘要             |
| --- | -------- | ----- | ----------------------- | -------- | ---------------- |
| 1   | {名称}   | {URL} | 官方文档/统计/案例/社区 | 高/中/弱 | {一句话关键信息} |
```

## 二、填充示例

以下为"结论先行"部分的填充示例，展示预期的具体程度：

```markdown
## 一、结论先行

- **总体判断**：该技术方案可行但收益被高估，建议缩小范围后再推进
- **综合评分**：5.5-6.0 / 10
- **关键发现**：
  1. 核心性能瓶颈确实存在，但根因不是语言选择而是架构设计
  2. Rust 重写的工程成本被低估约 3 倍（团队无 Rust 经验）
  3. 渐进式重写关键热路径可获得 80% 收益，仅需 20% 投入
- **最大风险**：团队 Rust 学习曲线导致项目延期超过 6 个月
- **核心建议**：先用 2 周做单模块 PoC 验证性能假设，再决定是否全面重写
```
