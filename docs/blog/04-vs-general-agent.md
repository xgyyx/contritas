# 如果你能用 Claude Code，为什么还要 Contritas？

> 本系列前几篇：
> 1. [《我做了一个会跟你唱反调的 AI 尽调 Agent——聊聊 Contritas 的架构与选型》](./01-architecture-and-stack.md)
> 2. [《怎么让 LLM 跟你唱反调——Contritas 是怎么把"反驳"写进 Prompt 的》](./02-anti-flattery-prompt-design.md)
> 3. [《Phase 4 交叉验证：让 LLM 找证据矛盾，而不是顺着事实线挑顺耳的写》](./03-phase4-cross-validation.md)

---

## 一、先把丑话说在前面

写一个垂直 Agent，最尴尬的问题是：

> "Claude Code / Cursor / Devin 都能联网搜、能拆任务、能写报告。我自己丢一个 prompt 进去不就完了？你这个 Contritas 凭什么存在？"

我不打算装作没听见这个问题。**单次、轻量、对结果纪律性要求不高的尽调，通用 Agent 大概能拿到 Contritas 70-80% 的报告质量。** 这是事实，不承认就是欺负读者没用过。

但这个数字背后有几个被四舍五入掉的细节，正好是 Contritas 的全部理由。这篇就把它拆开讲。

不讲营销口径。讲：**70-80% 这个数字怎么来的，剩下的 20-30% 在哪，以及——更重要的——模型再升级一轮，剩下的 20-30% 还守得住吗？**

---

## 二、通用 Agent 现在能做到的，先承认

给 Claude Code 一个写得够好的 prompt：

```
你是一个尽调 Agent。任务：
1. 把我的命题拆成 3-8 个核心假设
2. 每个假设找正反两面证据，至少 5 个独立来源
3. 标注证据可信度
4. 找证据矛盾点
5. 输出带置信度的报告，结论先行
```

它能跑出来。质量也不差。原因：

- **1M 上下文 + WebSearch / WebFetch 已经够用**，Phase 3 检索的核心能力 LLM 自己内置了一半
- **拆解、规划、综合是模型的母语能力**，6-Phase pipeline 更多是把 LLM 隐式做的事显式画出来
- **Claude 4.7 这一代模型本身的纪律性已经不差**——你叫它找反证，它真的会找

我自己测过：同一个命题，Claude Code + 仔细写的 prompt，输出和 Contritas 在**论点结构和主要风险识别**上能对上 70% 以上。

所以工程壁垒确实薄。这一点没什么好遮的。

---

## 三、那剩下的 20-30% 在哪

差距不在"能不能写出报告"，在**"每跑一次都能稳定地写出同等质量的报告"**。把它拆成五件具体的事。

### 3.1 反证纪律：靠 prompt 守不住

让 Claude Code 找反证，它第一轮认真找，第二轮就开始偷懒——找几条意思一下就回去综合了。这是 LLM 的固有问题：**确认偏误是模型的默认行为，不是 bug**。

Contritas 把反证焊成了流程：

- Phase 1 拆假设时，每个假设强制带 `counter_question`（如果它不成立，会因为什么）
- Phase 3 检索时，搜索关键词必须包含反向词（"X 失败案例"、"X 风险"、"X criticism"）
- Phase 4 交叉验证里，证据从入库就被打 `supports / weakens / qualifies` 三标签，少哪一类都会触发警告
- 报告自检阶段，如果一篇报告全是 `supports`，会被判定为"未充分质疑"，回退 Phase 3 重补反证

这套东西不是不能写进 prompt——但你得每次都写一遍，每次都检查模型有没有偷懒，每次都手动回退重跑。**Contritas 把"每次都"变成"自动"，这是产品和脚本的差别。**

具体怎么写进系统里的，详见 [《怎么让 LLM 跟你唱反调》](./02-anti-flattery-prompt-design.md)。

### 3.2 可审计的证据库 vs 一篇 markdown

通用 Agent 的输出形态是**一篇 markdown**：
- 引用是文中的链接
- 想知道某条结论的依据，得在文中翻
- 想换个角度重看一遍，得让它重写
- 想跨多次研究复用某条证据，没法做

Contritas 的输出形态是**结构化数据**：
- `evidence` 表存原始证据，每条带 url / 抓取时间 / 关系标签 / 可信度
- `cross_validations` 表存矛盾对，每对带具体两条证据 ID 和归因
- `report` 引用 `[E12]` 这种编号，点过去能看到原文
- 同一份证据可以被多次研究复用、可以被审计、可以被纠错

**这不是技术难点，是产品形态的选择**。但选择不一样，使用场景就不一样：通用 Agent 给你一篇文章，Contritas 给你一份能往上盖章的尽调底稿。

### 3.3 搜索源多样性：单一渠道的天花板很矮

Claude Code 的内置搜索就那一种。遇到付费墙、动态渲染页面、CDN 反爬、地区限制——直接断。

Contritas 的搜索层是降级链：

```
Tavily（主搜索，带摘要）
  ↓ 失败
Serper（备搜索，Google 直出）
  ↓ 拿到 URL，需要全文
Jina Reader（轻量提取，免费额度高）
  ↓ 失败 / 反爬
Firecrawl（浏览器渲染，付费但能搞定 JS-heavy 页面）
  ↓ 仍失败
Web Archive（兜底，至少有历史快照）
```

**这五层在不同场景下命中率差很多**。技术尽调里 Jina Reader 直接读 RFC / arxiv 没问题；商业尽调里很多公司官网做了反爬，必须 Firecrawl；时效不强但渠道封了的内容，Web Archive 反而救命。

通用 Agent 不是不能加这些 Provider，而是**它不会为某个垂直任务主动加**——它是通用的，所以默认配置必须是通用配置。

详见 [搜索层文档](../architecture/search-layer.md)。

### 3.4 长任务可恢复：陪跑 40 分钟是不可接受的

一次完整尽调跑完要 10-60 分钟。这个时长在通用 Agent 里非常痛苦：

- 用户要么挂在 chat 窗口前陪跑（注意力浪费），要么开新对话回头再看（上下文断了）
- 中途出错（API 限流、单次工具调用超时、模型 reasoning 跑飞），通用 Agent 一般是**整个对话作废重来**
- 跑到 80% 时想插一句"等等，把 Phase 3 那个维度多挖一点"，几乎不可能

Contritas 的设计基线就是为长任务做的：

- BullMQ + Redis 把研究放后台，前端 SSE 拿进度，关掉浏览器再开还在跑
- XState 状态机每个 Phase 后序列化一次到 DB，任何一步崩了都能从断点续做
- 用户可以中途追问、可以指定深挖某个维度、可以在报告生成后单独迭代某一段

**这部分纯粹是产品工程问题，不是 LLM 能力问题**。但产品工程问题在用户体验上的差异是数量级的。

### 3.5 非开发者可用：CLI 不是产品

这条最现实，也最容易被复刻——但**短期内**它确实是壁垒：

- Claude Code 是开发者工具，要会用 CLI、会写 prompt、会看 JSON
- Contritas 是 Web 产品，输入框 → 等结果 → 看报告 → 不满意点"深挖"

目标用户（独立创业者、早期投资人、小团队负责人）里**会写 prompt 调通用 Agent 的不超过 20%**。剩下的 80%，CLI 这个形态本身就把他们挡在门外。

## 四、然后，模型再升级一轮，这些差距还守得住吗？

诚实答案：**3.1 / 3.2 / 3.4 守得住，3.3 / 3.5 会越来越薄。**

- **3.1 反证纪律 / 3.2 证据库结构 / 3.4 可恢复长任务** —— 是产品形态和工程化的差异，不是模型能力的差异。模型再强也不会自动变成有 schema 的数据库
- **3.3 搜索源 / 3.5 非开发者形态** —— 一旦通用 Agent 平台原生集成更多 Provider、推出更友好的非 CLI 入口（这事正在发生），差距就会被压缩

所以如果 Contritas 的护城河只停留在上面这五件事，是会被时间稀释的。**真正想立住的护城河必须再往前一步**——这部分目前还在路上，但方向是清楚的：

### 4.1 领域 rubric 沉淀（最重要）

通用 Agent 的尽调是"通用方法 + 临时上下文"，而真正的人类尽调专家是"通用方法 + **特定行业的判断标准**"。

举几个例子：

- **PE 尽调**看一家公司，会盯现金流质量、应收账款账龄、关联交易、对赌条款——这些东西通用 Agent 不会主动盯，因为它没有 PE rubric
- **学术综述**评价一个领域结论，会看引用网络、复现失败率、利益冲突声明——通用 Agent 默认只看 abstract
- **技术选型**评价一个框架，会看 issue 关闭率、major contributor 流失、breaking change 频率——通用 Agent 只会去找官网吹牛页

Contritas 后面要做的事：**把这些 rubric 写成可插拔的"研究模板"**——用户选"PE 尽调"或"技术选型"，Phase 1 的拆解模板、Phase 3 的来源白名单、Phase 4 的矛盾归因规则都跟着切换。

这件事通用 Agent **理论上**也能做，但现实里它不会做——因为它的产品定位就是"对所有任务都通用"。**通用是它的核心承诺，也是它放弃垂直深度的原因。**

### 4.2 跨研究的证据复用

每一份尽调都积累几十条证据。如果这些证据只服务于一次报告，那就太浪费了：

- 同一家公司被研究过两次，第二次完全可以复用第一次的工商记录、新闻、财报
- 同一个行业的"常见反证"是高度重叠的，可以沉淀成"该行业的尽调 checklist"
- 用户的多次研究构成一张知识图谱，可以发现命题之间的关联（"你研究的 A 公司和上次研究的 B 公司有共同投资人"）

**这是数据沉淀的护城河，不是 LLM 的护城河**。通用 Agent 没有数据库（它不存东西），所以这件事天然做不到。

### 4.3 把"产品形态"做成默认

3.5 单独看不算护城河，但把 3.5 + 4.1 + 4.2 绑在一起，就形成了"非开发者也能用 + 行业 know-how 内置 + 历史研究自动复用"的产品组合——这个组合通用 Agent 短期内不会跟，因为它和"通用"两个字直接冲突。

---

## 五、结论

回到开头那个问题：**如果你能用 Claude Code，为什么还要 Contritas？**

诚实地分场景：

| 你的情况 | 推荐 |
|----------|------|
| 一次性、轻量的研究问题，自己会写 prompt | **用 Claude Code 就够了**，不需要 Contritas |
| 重要决策前的尽调，需要可审计、可复盘的证据底稿 | Contritas 的反证纪律 + 证据库结构有用 |
| 跑完一次还想深挖、迭代、追问，不想从头再来 | Contritas 的长任务可恢复架构有用 |
| 自己不写代码，想给团队/合伙人/家人用 | Contritas 是产品，Claude Code 是工具 |
| 你做的是某个特定行业（PE / 技术选型 / 政策研究）的反复尽调 | **未来**的 Contritas 会有针对性的 rubric——现在还没完全做完，但这是方向 |

我不打算说 Contritas 在每个维度上都碾压通用 Agent。**那是营销话术，写在博客里读者会笑。**

我说的是：**通用 Agent 的边界在"通用"两个字，而尽调这件事的最高水平不在通用——它在垂直纪律、可审计输出、和领域 know-how 的沉淀**。Contritas 赌的是后者，承认前者。

这个赌局现在还没到拍板的时候。但至少方向是清楚的：

> 不和通用 Agent 比 LLM 能力——比不过，也不该比。
>
> 比的是承诺：**保证不放过你的盲点**。
