## 一、最近的 Agent 产品在比谁的"思考流"更花哨

打开 Devin、Manus、ChatGPT Agent、Operator 这些产品，UI 上都有同一个东西：

> **左半边屏幕在演示 Agent 在干什么**——它打开了什么网页、滚动到哪、点了哪个按钮、写了几行 Python、改了哪个文件。
>
> 你坐在右半边，像看一部慢镜头默片。

这种 UI 形态正在成为 AI Agent 产品的"行业默认"。它有几个看起来很合理的好处：

- **显得透明**：用户看得到每一步，建立信任
- **显得智能**：滚动浏览的动画、代码在屏幕上一行行打出来——很有"它在认真干活"的氛围
- **显得不一样**：和 ChatGPT 的纯文字对话拉开差距，是产品的差异化卖点

Contritas 没做这种 UI。技术上做不到吗？不是——SSE 通道每秒能推几十个事件，把"Agent 现在在搜什么 query、抓哪个 URL、想什么"都流式渲染到前端，纯属一周工作量。

**是产品决策**：用户**不该**看到 Agent 在干什么。用户应该看到的是结论怎么演化、还要等多久、有没有死。

这篇讲清楚为什么。

---

## 二、看 Agent "思考"看上 30 秒就开始疲倦

我自己用过那几款"思考流"演示型 Agent，发现一件事：

- **前 30 秒**：哇，它真的在搜，真的在打开 Reddit，看起来好聪明
- **30 秒到 5 分钟**：开始走神，刷手机，回头瞄一眼"它还在搜"
- **5 分钟之后**：完全不看 UI 了，等结果就行
- **结果出来时**：回头看左半边屏幕的"演示"——已经完全不重要了，**你只在乎右边的产出**

这是一个显著的体验衰减曲线。前 30 秒的"哇"是产品演示视频效果；真正的使用者过几次之后，根本不会盯着思考流看。

Contritas 的尽调一次跑 10-60 分钟。如果用户被诱导坐在屏幕前盯着"它现在搜到第 17 个网页了"——这是巨大的**注意力浪费**：

- 用户的时间不该用来当观众
- 用户的耐心是一个有限资源，不应该消耗在"看 Agent 干活"这种没产出价值的事情上
- 一旦用户开始厌倦"看 Agent 思考"，他对产品的整体好感都会下滑

**所以"演示思考"这种 UX 形态，在长任务里是负效用的**。它对短任务（30 秒-2 分钟）有营销价值，对长任务（10 分钟以上）就是负担。

## 三、用户真正需要的进度信号，只有三类

那用户在等待期间，到底需要什么？我自己跑了几十次研究后总结，**有用的进度信号其实只有三类**：

### 3.1 流程在哪一步

不是"Agent 现在搜了 X 个网页"，是"现在在 Phase 几"。这个粒度足够回答"还在跑吗 / 进度大概多少"。

### 3.2 有没有死

最重要的信号——**确认它没卡死**。研究跑十几分钟用户最大的焦虑是"是不是卡了 / 是不是网络问题 / 我要不要刷新"。

### 3.3 大概还要多久

ETA。这个信号让用户决定"我去喝杯咖啡 / 我去吃个饭 / 我去睡觉等结果"。

**这三件事就是全部**。不需要看到 Agent 在搜什么 query、不需要看到它在读哪个网页、不需要看到它在想什么。

## 四、Contritas SSE 的 7 类事件，对应的就是这三件事

看 `packages/shared/src/types/events.ts`，Contritas 推 7 类事件：

```typescript
export type ProgressEvent =
  | PhaseChangeEvent           // 流程在哪一步
  | DimensionUpdateEvent       // 当前维度搜了几轮、收了多少证据
  | SearchExecutedEvent        // 一次搜索完成（有进展信号）
  | EvidenceAddedEvent         // 一条证据入库（有进展信号）
  | ErrorEvent                 // 出错了，可恢复 / 不可恢复
  | EtaUpdateEvent             // 大概还要多久
  | ReportReadyEvent;          // 报告好了
```

把它们映射回三类信号：

| 用户需要的信号 | 对应的事件 |
|---------------|-----------|
| 流程在哪一步 | `phase_change` / `dimension_update` |
| 有没有死 | `search_executed` / `evidence_added` / `error` |
| 大概还要多久 | `eta_update` |
| 完成 | `report_ready` |

注意几个**没**做的：

- ❌ 没有 `agent_thinking`（推模型 reasoning trace）
- ❌ 没有 `page_visited`（推 Agent 打开了什么网页）
- ❌ 没有 `query_planned`（推 Agent 想出了什么搜索词）
- ❌ 没有 `tool_call`（推 Agent 调了什么工具）

这些都是"演示思考流"会推的事件，Contritas 一律不推。

### 4.1 search_executed 和 evidence_added 不是给用户"看"的

注意上面表里把 `search_executed` 和 `evidence_added` 归到"有没有死"——不是"流程在哪一步"。这是有意的。

这两个事件的产品意图**不是给用户看 Agent 在搜什么**，是**让用户感觉到它还在动**。前端拿到这两个事件应该做的是：

- 增加一个累积计数器（"已检索 23 次 / 已收集 47 条证据"）
- 不显示具体 query 内容、不显示具体 URL

用户看到累积数字在涨 = "它还在干活"。这就够了。**不需要把每条具体 query 渲染出来——那是噪音**。

### 4.2 ETA 是 Contritas SSE 设计里最该被夸的事件

`eta_update` 在大多数 Agent 产品里都没有。原因是 ETA 不好做：状态机没跑过怎么估算剩余时间？

Contritas 的做法是基于**历史 phase 平均时长**做加权预估，每完成一个 phase 就重新算一次推给前端。这个数字不绝对准，但有它和没它对用户的体验差距是**质变的**：

- 没 ETA：用户每分钟都焦虑一次"还要多久"
- 有 ETA：用户看到"大概还要 8 分钟"，关掉浏览器去吃饭，时间到回来看结果

**ETA 这个事件比所有"演示思考"加起来更有产品价值**。但它没法做成炫酷的演示视频，所以没人吹。

## 五、产品哲学的根：审计口径 vs 表演口径

到这里其实可以把这件事抽象到更高一层。

Devin / Manus / Operator 那种"演示思考流"的 UI，本质上是**表演口径**：

- 让用户觉得 Agent 在"认真思考"
- 让用户觉得自己买的不是黑盒，是有透明度的工具
- 让产品演示视频有视觉冲击力

但 Contritas 的整个产品定位是 **审计口径**——

- 不证明你是对的，证明你经得起尽调
- 不让你看 Agent 怎么想的，让你看证据怎么追溯的
- 不演示过程，交付可审计的产出

**这两种口径在 UX 上是冲突的**。表演口径希望用户全程在场，因为这是它价值兑现的地方；审计口径希望用户离场，因为它的价值在最后那份报告，不在过程。

强行在 Contritas 上加"思考流演示"，会发生一件很尴尬的事：

> 用户看了 30 秒思考流，觉得"挺酷"，然后等 10 分钟报告出来，发现报告上不认识的来源、看不懂的评分、还有几个"证据不足"的标注——这时候之前看的 30 秒思考流**没有任何帮助**他理解最终输出。它只是消耗了用户的注意力，没回报。

而审计口径的 UI 要做的事是：

> 让用户在等待时知道"它没死、大概还要多久"，然后把所有注意力留到最终报告——在那里铺开**证据追溯链、评分双向解释、矛盾标注**——用户读报告的 5 分钟，比之前看 30 秒思考流的价值高一个数量级。

**这是两种产品哲学的根本冲突**。Contritas 选了后者。

## 六、未来会不会做思考流？

未来一种**有限的**"思考可见性"是可能加的，但和"演示思考流"不一样：

### 6.1 可能加的：审计性思考可见

研究完成后，用户**主动**点开某条结论，能看到：

- 这条结论用了哪几条证据（已经做了）
- 这几条证据是从什么 query 搜出来的（部分做了）
- 当时 LLM 评估这些证据的简要 reasoning（**这部分还没做**）

这是"事后可审计"——用户想深究的时候能查到，但不会强迫他在过程中看。

### 6.2 不会加的：实时演示

实时把"Agent 在打开什么网页 / 在想什么"流式推给前端、配合炫酷的滚动动画——这个永远不会做。

理由就是上面这一整篇。**它和 Contritas 的产品哲学根本冲突**。

### 6.3 能折中的：高级用户的"详细模式"

可能加一个用户偏好开关——默认 UI 极简（三类信号），但开发者/重度用户可以打开"详细事件流"看到所有 SSE 事件原文。

这个方向是合理的：默认体验保持审计口径，给少数想深挖的用户开一道侧门。

## 七、收尾

很多 AI Agent 产品在 UX 上的"创新"，本质上是把过程演示当成了产品本身。这在短任务（生成图片、查个事实、写段代码）里有效，因为过程短到能当娱乐看；在长任务（10 分钟以上的真活）里，它**消耗的是用户最稀缺的资源——注意力**，而没有等价回报。

Contritas 选的相反路径很简单：

> **过程要让用户能离开，结果要让用户能审计。**

SSE 的 7 类事件、ETA 推送、累积计数——这些都是为了"让用户能离开"。
报告里的证据追溯链、评分双向解释、自检失败标注——这些都是为了"让用户能审计"。

中间那一整段"Agent 在干什么"，**对用户来说不重要**——只对产品演示视频重要。

而 Contritas 不打算做演示视频。
