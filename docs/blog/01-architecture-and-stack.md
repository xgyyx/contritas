# 我做了一个会跟你唱反调的 AI 尽调 Agent —— 聊聊 Contritas 的架构与选型

> 不证明你是对的，证明你经得起尽调。

## 一、起因：所有 AI 都太"听话"了

一个独立创业者朋友最近在做出海 SaaS。他用了大半个月的 ChatGPT Deep Research、Perplexity、各种 AI 助手做调研，得到的结论惊人地一致——**项目可行、市场广阔、增速强劲**。

然后他问我："为什么没有一份调研告诉我这事可能不成？"

这就是 AI 调研工具今天的通病：它们是**搜索增强问答**，你问什么它答什么；你想被支持，它就找证据支持你。这个口径在很多场景没问题，但在**做决策**这件事上是反常识的——专业尽职调查的口径从来不是"鼓励"，而是"审计"。

银行做尽调不是为了告诉你这个项目多好，是要找出它哪儿不靠谱。

我想做一个反过来的 Agent：你给它一个判断、一个想法、一个论点，它**主动从反面拆**，然后告诉你哪些假设最脆弱、证据最薄、风险最大。

这就是 Contritas。名字来自拉丁文 _Contra_（反向）+ _Veritas_（真理）—— **通过反证逼近真相**。

## 二、产品上做了什么

一句话定义产品：**输入一个决策命题，输出一份带置信度的尽调报告**。

整个流程是一条 6 阶段流水线：

```
Phase 0  输入验证   → 这个命题足够具体吗？模糊就追问
Phase 1  假设拆解   → 把命题拆成 3-8 个核心假设，按"如果这个不成立影响多大"排序
Phase 2  研究规划   → 为每个假设规划研究维度、搜索关键词
Phase 3  多源检索   → 中英双语、多轮检索、抓页面正文（不只看摘要）
Phase 4  交叉验证   → 同一假设的多源证据是否一致，矛盾必须标注
Phase 5  报告综合   → 结论先行 + 置信度评分 + 行动建议
```

跟"问答"最大的区别是 Phase 1 和 Phase 4：

- **拆假设而非答问题**——用户输入"我用 Rust 重写核心服务值得吗"，它不会去搜"Rust 优势"，会先拆出"性能瓶颈真的在语言层"、"团队学习成本可控"、"生态成熟度足够"等假设，然后**针对每个假设找反证**。
- **交叉验证作为硬约束**——同一假设有 2 条以上方向矛盾的证据，必须标注矛盾、分析口径差异，而不是挑顺耳的那条写进结论。

报告每条结论都要回引到具体证据编号，**不允许出现无来源支撑的断言**。

## 三、技术上踩过的几个关键坑

下面进入正题。讲讲为什么是这套架构、每个选型背后的纠结。

### 3.1 第一个坑：研究是个"长任务"

一份高复杂度报告要 30-60 分钟，涉及 80-150 次搜索、几十次 LLM 调用。这个时长决定了一件事：**绝对不能把它绑在 HTTP 请求里**。

请求超时、客户端断网、Worker 重启……任何一个都能让用户的活儿白干。所以一开始就分成三个进程：

```
Hono API (port 4000)  → 接请求、写 DB、推到队列、立刻返回 202
BullMQ Worker         → 从队列拿任务，跑完整 6 阶段流程
Next.js Web (port 3000) → 通过 SSE 订阅进度
```

队列用 **BullMQ + Redis**，有几个不能让步的能力：

- **进度追踪** —— Worker 把每个 Phase 的中间状态写回，客户端实时看
- **重试** —— 搜索 API 抖一下就重试，不让整个研究失败
- **断点续做** —— Worker 崩了，下次拉起从最后一个完成的 Phase 续

### 3.2 第二个坑：多阶段流程怎么写不糟心

第一版用 if/else + switch/case 写工作流，写到 Phase 4 心态就崩了——并行的维度检索、自检失败回退 Phase 3、等待用户回复追问……状态稍微多一点，分支就开始嵌套。

最后切到 **XState v5 状态机**。换上之后几个能力直接到位：

| 能力 | XState 给的 |
|---|---|
| 并行状态 | Phase 3 各维度并行检索是天然的 parallel state |
| 条件转换 | 自检不通过 → 回退 Phase 3，guard 一行表达 |
| 等外部输入 | Phase 0 追问用户后挂起，等 USER_RESPONSE 事件 |
| 序列化恢复 | snapshot 序列化进 DB，Worker 拉起从快照恢复 |
| 可审计 | 状态转换历史天然有记录，调试和复盘都靠它 |

XState 不是没成本——学习曲线在那、心智模型要先对齐——但对于**步骤多、有分支、可能中断**的工作流，这成本值得花。

### 3.3 第三个坑：LLM 成本会失控

第一次跑通端到端，看了一眼账单——一份高复杂度报告 LLM 用掉将近 1 美金。乘上潜在的并发量，这个数字撑不住。

拆开看 token 流向，发现一件事：**不是所有 Phase 都需要顶级模型**。

- Phase 1（拆假设）、Phase 4（交叉验证）、Phase 5（综合报告）—— 高难度推理，必须用强模型
- Phase 3 里的"从一篇文章里提取关键摘录"——这是体力活，便宜模型完全干得了

于是抽象了一层 **Model Router**，按 Phase 路由到不同 Provider/模型：

```
inputValidation     → 中等模型，判断力够即可
decomposition       → Claude Opus（拆得不好后面全完）
planning            → 任意能写结构化输出的
evidenceExtraction  → DeepSeek / GPT-4o-mini（高频低成本）
crossValidation     → Claude / GPT-4o
synthesis           → Claude（长输出 + 行文质量）
```

底层是 **Adapter 模式**：Claude SDK、OpenAI Compatible（兼容 one-api / litellm / DeepSeek / vLLM 等任何 OpenAI 协议端点）背后都是同一个 `LLMProvider` 接口。换 Provider 不用动业务代码，只改一行环境变量。

加上搜索结果缓存（24h）、相同输入的 LLM 响应缓存（7 天）、Token 预算上限（耗尽进入 `budgetExceeded` 终态而不是无限烧钱），单次研究成本压到了可接受区间。

### 3.4 第四个坑：搜索质量是天花板

LLM 再强，喂垃圾进去也只能输出垃圾。早期版本只接了 Tavily，碰到中文长尾内容覆盖就拉胯。

最后定型的搜索栈是**两层降级链**：

```
搜索：    Tavily (主) → Serper (备)
内容提取：Jina Reader → Firecrawl → Web Archive
```

提取层的设计有个值得说的细节——**接受部分缺失**。某个页面反爬严格、JS 渲染抓不到？跳过，记录一条"提取失败"。报告里允许标注"证据不足"，**不允许伪造证据**。这是产品原则也是工程纪律：宁可少证一条，不可错证一条。

### 3.5 第五个坑：进度怎么推给前端

用户提交后要等几十分钟。这段时间里如果界面只转个圈，用户大概率以为卡死了，10 秒不到就关掉。

进度推送选型纠结过 SSE vs WebSocket，最后选了 **SSE**：

| 维度 | SSE 占的便宜 |
|---|---|
| 方向 | 单向，服务端推就够（用户操作走 REST 即可） |
| 重连 | 浏览器原生 EventSource 自动重连 + `Last-Event-ID` 补发事件 |
| 基础设施 | CDN / 反代直接通，不需要 sticky session |
| 实现 | Hono 内置 `streamSSE`，几行代码 |

事件本身定义成一组结构化类型：`phase_change` / `dimension_update` / `evidence_added` / `eta_update` / `report_ready` / `error`。前端用一个 React Hook 订阅，配 Zustand 累积事件，拼出一个不断生长的进度视图。

事件全量写入 **Redis Stream**，断线重连用 `XRANGE` 补发，7 天 TTL 自动清。

## 五、几个看似无聊但很重要的选择

下面这些不是"亮点"，但每一个都是踩过坑后才确定的。

**Monorepo + pnpm + Turborepo**。前后端共享 TypeScript 类型，Zod schema 也共享——前端校验过的字段后端不用再写一遍校验逻辑。Turborepo 缓存让 CI 在没改动的 package 上零开销。

**Drizzle 而非 Prisma**。原生 SQL 思维 + 完整 TS 推断 + 零运行时开销。JSONB / 数组字段支持比 Prisma 干净，迁移文件是普通 SQL 看得懂。

**Zod 作为前后端契约**。所有 API 请求/响应、所有 LLM 结构化输出都过 Zod。LLM 输出 schema 不合规直接重试，不让脏数据进数据库。

**ULID 而非 UUID**。可排序、可读性更好，在数据库里建索引也更友好。

**Hono 而非 Express**。TypeScript 原生、内置 SSE、性能好、跨 Runtime（Node / Bun / Edge 想换就换）。

## 七、我从这个项目里学到的

做完这一轮我有两个挺强的体感：

**第一，AI 应用工程的难点正在从"能不能跑"迁移到"成本和可靠性"。** 让 LLM 跑通一个 demo 已经是几十行代码的事，但要让一个长任务、多阶段、多 Provider 的 AI Agent 在 60 分钟里不崩、不超支、可恢复、可审计——这才是工程量。

**第二，AI 工具最容易被忽略的设计是"敢不同意用户"。** 大部分产品在迎合用户预期上做得太多了，以至于失去了它本该提供的价值。Contritas 想验证一件事——一个**结构性地反对你**的 AI，对决策者更有用。
