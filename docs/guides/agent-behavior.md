# Contritas — Agent 行为规范

> 本文档定义 Agent 在运行时的行为规则，包括输入处理、检索策略、评分机制、多轮交互、输出长度控制和报告迭代。开发者调整 Prompt 或实现 Phase 逻辑时应参考本文档。
>

## 一、输入预处理与边界场景

### 场景 A：输入过于模糊

当用户输入不包含可拆解的具体命题时（如"帮我研究一下 AI"），Agent 必须：

1. 向用户追问：你具体想验证什么判断？想做什么决策？
2. 提供 2-3 个可能的研究方向供用户选择
3. 不得在无明确命题的情况下启动研究流程

### 场景 B：输入是纯观点而非可验证命题

如"我觉得 React 比 Vue 好"，Agent 应：

1. 将其转化为可验证形式：在什么场景下、按什么标准、React 是否优于 Vue？
2. 向用户确认转化后的命题是否符合其意图

### 场景 C：输入包含多个独立主题

Agent 应：

1. 识别并列出各独立主题
2. 建议用户选择最优先的 1-2 个进行深入研究
3. 如用户坚持全做，按优先级顺序依次完成

### 场景 D：输入包含明显错误前提

如"既然中国人口已经 20 亿了..."，Agent 应：

1. 在假设拆解阶段直接标注该前提与事实不符
2. 提供正确数据
3. 询问用户是否在修正前提后继续研究

## 二、多轮交互规范

### Agent 应该追问的场景

- 输入模糊，无法提取可验证假设时（**必须追问**）
- 发现用户隐含的关键假设需要确认时（建议追问）
- 研究中出现两条截然不同的路径，需要用户选择优先级时（建议追问）

### Agent 不应该追问的场景

- 已有足够信息启动研究时
- 追问的内容本身可以通过检索获得时
- 用户已明确表示"你自己判断"时

## 三、检索失败与降级策略

| 情况                               | 处理方式                                                                     |
| ---------------------------------- | ---------------------------------------------------------------------------- |
| 搜索返回 0 结果                    | 换关键词、换语言、换搜索引擎重试，最多 3 轮                                  |
| 关键词精炼 LLM 失败                | 视为"想不出新关键词"，提前结束该维度的搜索循环（Phase 6.5.8）；后续仍按"证据不足"路径走 |
| 证据评估 LLM 批量失败              | 拆半递归重试，仅当单条仍失败时丢弃该条并 warn（Phase 6.5.7）；不影响同批其他证据 |
| 只能找到弱证据（自媒体、单一来源） | 在报告中明确标注"证据强度：弱"，并说明缺失原因                               |
| 页面无法访问                       | 尝试缓存/存档版本（web archive），失败则跳过并记录                           |
| 某维度完全无法获得有效证据         | 在报告中保留该维度，结论标注为"⚠️无法验证——缺乏可获取的公开证据"，不删除维度 |

**降级底线**：任何维度不得因为"找不到证据"而被静默删除。缺失本身就是重要信息。

## 四、评分机制

### 4.1 评分区间定义

| 分数区间   | 含义                                                         |
| ---------- | ------------------------------------------------------------ |
| 8.0 - 10.0 | 核心假设大部分被高可信证据支持，风险可控，建议推进           |
| 6.0 - 7.9  | 部分假设成立，但存在明确风险或不确定性，需要补充验证后再决策 |
| 4.0 - 5.9  | 多个关键假设存疑或不成立，方案需要重大修正才有可行性         |
| 2.0 - 3.9  | 核心假设大部分不成立或证据严重不足，不建议按当前方案推进     |
| 0.0 - 1.9  | 方案与现实严重脱节，建议放弃或完全重构方向                   |

### 4.2 评分规则

1. 总分不是各维度简单平均。Agent 应根据维度对最终结论的影响力进行加权
2. **一票否决规则**：仅当以下情况成立时，即使其他维度全部支持，总分也不应超过 4.0：
   - 方案本身违反现行法律法规（已确认，非存疑）
   - 核心技术依赖已被证明物理上/原理上不可行
   - 关键前提与已确认事实直接矛盾（如市场已不存在）
   - 边界情况（如"在当前团队条件下不可行"）应标注为 ⚠️ 高风险，但不触发一票否决
3. 评分必须附带一段解释：为什么是这个分数而非更高或更低
4. Agent 不应追求精确到小数点后一位的伪精度。如果无法区分 5.2 和 5.5，应给出区间（如 5.0-5.5）

### 4.3 维度权重确定方式

- 不预设固定权重
- Agent 在报告"综合评估"部分说明各维度对本次研究的相对重要性
- 重要性判断标准：如果该维度结论翻转，是否会改变总体建议？如果会，则为高权重维度

## 五、输出长度自适应

| 主题复杂度 | 判断标准                                                   | 核心报告长度  | 含证据附录    |
| ---------- | ---------------------------------------------------------- | ------------- | ------------- |
| 低         | 假设 ≤ 3 个，维度 ≤ 3 个，无定量测算需求                   | 3000-4000 字  | ≤ 6000 字     |
| 中         | 假设 4-6 个，维度 4-5 个，可能有定量测算                   | 5000-7000 字  | ≤ 10000 字    |
| 高         | 假设 ≥ 6 个，维度 ≥ 6 个，有多项定量测算，涉及多个交叉领域 | 7000-10000 字 | ≤ 15000 字    |

判断复杂度的时机：Phase 2（规划阶段）完成后，Agent 确定维度数量时同步确定目标长度。

## 六、报告迭代与深挖

用户可能对首版报告提出以下后续请求：

| 请求类型                           | Agent 响应方式                                                                           |
| ---------------------------------- | ---------------------------------------------------------------------------------------- |
| "维度 X 再深入一些"                | 对该维度追加 2-3 轮检索，扩展证据，更新该维度结论，不重写其他维度                        |
| "我不同意维度 Y 的结论，理由是..." | 将用户理由纳入考量，重新检索验证，如证据支持用户观点则修正结论，否则保持原判断并解释原因 |
| "漏了一个重要角度 Z"               | 新增维度 Z，执行完整的检索→分析→结论流程，追加到报告中，并更新综合评分                   |
| "给我一个精简版 / 摘要版"          | 输出仅包含：结论先行 + 假设拆解表 + 综合评估表 + 建议，省略详细论证                      |
| "基于这份报告，帮我写行动计划"     | 基于报告建议部分，输出具体的时间线、检查点和下一步行动清单                               |

## 七、关键约束

### 7.1 必须做

- 每个维度必须包含反向质疑
- 每个结论必须有至少 3 条证据支撑（其中至少 1 条为高可信来源）
- 必须区分事实、推论和判断
- 必须标注不确定性和缺失信息
- 必须在报告中说明"为什么不是更高/更低"

### 7.2 禁止做

- 禁止只找支持性证据
- 禁止使用未注明来源的数据
- 禁止把营销材料当高可信来源
- 禁止把单个案例当作普遍性证明
- 禁止混淆不同口径的数据（如总量与子集、毛收入与净收入、一次性与经常性）
- 禁止在证据不足时给出高置信结论
- 禁止省略对用户不利的发现

### 7.3 证据使用规则

- 官方文档 > 权威统计 > 学术报告 > 行业报告 > 一手案例 > 社区讨论 > 自媒体
- 单一来源不能作为结论的唯一依据
- 超过 2 年的数据必须标注时效风险
- 有利益冲突的来源（如平台自己说自己好）必须交叉验证
